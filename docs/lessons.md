# Project Lessons: lm-azuresentinel-pipeline

Project-specific learned behaviors. Global lessons live in `~/.claude/docs/lessons.md`.

Append to this file at session close-out. Each lesson: one-line rule, **Why:** line (incident/reason), **How to apply:** line (when it kicks in).

## LogicMonitor integration

- **LM Custom HTTP Delivery auth: leave both "Authenticate using" radios unselected and put auth in "Use custom headers".** **Why:** The LM UI presents Basic Auth and OAuth 2.0 as the two radio options, but neither is correct for Event Hub SAS. The "Use custom headers" toggle replaces them — custom headers carry the `Authorization` header directly. **How to apply:** For any external API that expects a non-Basic, non-OAuth `Authorization` scheme (SAS tokens, Bearer tokens, custom signatures), toggle "Use custom headers" ON and leave the auth radios empty. Confirmed in LM docs (https://www.logicmonitor.com/support/custom-http-delivery) and via live Event Hub test on 2026-04-20.
- **Event Hub's SAS token parser is strict about whitespace.** **Why:** `SharedAccessSignature   sr=...` (multiple spaces) produces `401 MalformedToken` even though the token itself is valid. The `&` separators inside the token are NOT the problem. **How to apply:** When pasting SAS into the LM form, select all, delete, paste clean. Copy-paste from terminals sometimes introduces extra whitespace. Before diagnosing token content, check that the separator between `SharedAccessSignature` and the parameters is exactly one space.
- **LM webhook test alerts use placeholder values, not substituted tokens.** **Why:** When you click "Test Alert Delivery" in the LM integration form, LM sends a payload with `alertId="LMD0"`, `severity="ok"`, `status="test"`, `startEpoch="YYYY-MM-DD HH:MM:SS GMT"` (a formatted string, NOT epoch seconds), `clearEpoch="Alert Clear Value"` (literal placeholder text), etc. **How to apply:** The transform must handle both formatted datetime strings AND integer epoch seconds for `startEpoch`/`clearEpoch`, AND treat literal sentinel strings like `"Alert Clear Value"`, `"##CLEARVALUE##"`, `"##START##"` as null. Real alerts use different formats than test alerts — do not design a parser based solely on the test payload.
- **LM escalation chains route to integrations via a User, not directly.** **Why:** LM's recipient model is `(User, Contact Method)` pairs. An integration becomes a recipient only if some user has that integration configured as one of their contact methods. Confirmed in LM docs (https://www.logicmonitor.com/support/escalation-chains) and via live creation 2026-04-20. **How to apply:** When automating escalation chain creation, the destination must be `{"type": "admin", "addr": "<username>", "method": "<integration_name>"}`. The `method` is the integration's **display name** (string), NOT its integer ID. `INTEGRATION` is not a valid recipient type even though the LM UI makes it look like one.

## Azure Sentinel / Logs Ingestion

- Modern Sentinel onboarding via `Microsoft.SecurityInsights/onboardingStates` still produces a legacy `Microsoft.OperationsManagement/solutions` resource named `SecurityInsights(<workspace>)`. **Why:** Internal implementation still creates the solution artifact for backwards compatibility. **How to apply:** When verifying Sentinel onboarding via `az resource list`, the solution resource IS the evidence — don't interpret its presence as legacy/incorrect provisioning.
- App Insights creates an implicit `microsoft.alertsmanagement/smartDetectorAlertRules` resource (`Failure Anomalies - <appinsights-name>`) automatically when a component is provisioned. **Why:** Default smart-detection rule, benign. **How to apply:** Expect resource count to be N+1 where N is what Bicep declares. Don't flag it as drift.
- Logs Ingestion API has 2-5 min ingestion latency for custom tables. **Why:** DCR processing and table population is asynchronous. The `upload()` call returns success immediately but the row doesn't appear in KQL queries until the ingestion pipeline flushes. **How to apply:** E2E smoke tests must poll the target table with at least 10 minutes of patience. Don't treat a 2-minute "no row found" as a pipeline failure — that's normal steady-state.
- `azure-monitor-ingestion` column-name matching is case-sensitive AND exact. **Why:** The DCR stream declaration's column names must match the `to_dcr_row()` keys byte-for-byte. Mismatched casing or typos silently drop the row (no error from the SDK, no error from the DCR — just nothing in the custom table). **How to apply:** Keep the DCR `columns` array (in `dcr.bicep`) and the `NormalizedAlert` field names in adjacent files. When changing the schema, update both and re-run the smoke test.

## Event Hub + Function

- `func azure functionapp publish` fails on WSL/Ubuntu-slim with ICU/globalization errors unless `DOTNET_SYSTEM_GLOBALIZATION_INVARIANT=1` is exported. **Why:** The .NET runtime inside `func` requires ICU libraries for culture-aware JSON deserialization; WSL default rootfs lacks them. **How to apply:** Always set `DOTNET_SYSTEM_GLOBALIZATION_INVARIANT=1` before invoking `func publish` on Linux dev environments. Deploy script does this automatically.
- `az functionapp deployment source config-zip` can return a false `ResourceNotFound` error immediately after Function App creation even when `az functionapp show` succeeds on the same resource. **Why:** The SCM-site endpoint that `config-zip` hits propagates separately from the Microsoft.Web/sites resource provider; there is a several-minute gap. **How to apply:** Prefer `func azure functionapp publish --build remote` with ICU workaround for deploys immediately after `az deployment group create`. Zip deploy via az CLI only reliably works after a several-minute warm-up.
- Event Hub trigger binding uses `%ENV_VAR_NAME%` syntax in the decorator to resolve the hub name at trigger attach time (e.g. `event_hub_name="%EVENT_HUB_NAME%"`). **Why:** Python decorators evaluate at module import, before app settings are guaranteed to be in `os.environ` for the host process. `%...%` is the Azure Functions binding expression that resolves at host startup. **How to apply:** Use `%...%` for any binding property that must read an app setting. Never read `os.environ` inside `@app.event_hub_message_trigger(...)` decorator arguments.
- Azure Oryx remote build CDN is an unreliable dependency. **Why:** `func azure functionapp publish --build remote` depends on Oryx pulling Python SDKs from `oryx-cdn.microsoft.io`. The CDN had multiple minute-long outages on 2026-04-20 during the POC build, failing each deploy with `TaskCanceledException`. **How to apply:** When remote build fails, fall back to local-wheel build: `pip install --target=function/.python_packages/lib/site-packages -r requirements.txt --platform manylinux2014_x86_64 --only-binary=:all: --python-version 3.12`, then `func azure functionapp publish <name> --python --no-build`. On x86_64 Linux, local build is actually faster than Oryx when it works.
- Azure Function v2 Python model requires `function_app.py` at the root of the deployed package. **Why:** The runtime looks for the `FunctionApp` instance by convention at the package root. Moving it into a subpackage breaks trigger discovery. **How to apply:** Keep `function_app.py` at the root; put business logic in `src/` and import from `function_app.py`. Don't try to restructure the entry point.
- Bicep `az deployment group create` can disturb a deployed Function's active code. **Why:** Redeploying the Function App resource via Bicep causes the Functions host to restart and may roll back to a cached or earlier version of the code package. Session 2 (2026-04-20) saw the Sentinel rule deploy cause the Function to revert from full-transform code to an earlier intermediate version. **How to apply:** After any Bicep deployment that touches the Function App module, re-run `func publish` to ensure the latest code is active. Always verify with a smoke test after infra redeploys.

## LM MCP usage patterns

- **LM MCP has no `create_http_delivery_integration` tool.** **Why:** The MCP surface does not cover Custom HTTP Delivery integration creation — only `get`-style queries on integrations and the downstream escalation chain / alert rule primitives. **How to apply:** For any LM→external-system integration automation, plan on a one-time manual UI step for integration creation. Automate only chain + rule afterward. See `docs/mcp-feedback.md` for the full list of gaps to request from the MCP maintainers.
- **LM MCP `create_escalation_chain` destinations schema is undocumented.** **Why:** The tool description says "Destination stages for the escalation chain" but gives no shape example. The working shape requires nested arrays (`destinations[].stages[][]`) with lowercase `"admin"` type and the integration's display name as `method`. **How to apply:** Until the MCP description is improved, fall back to UI creation for escalation chains that route to integrations. For chains that route to users (email), direct automation is fine.
- **When MCP tool response echoes stored state, use that as schema discovery.** **Why:** `create_alert_rule` succeeded and its response JSON contained the escalation chain's stored destination structure. That revealed the correct recipient schema (lowercase `admin`, `method` = integration name). **How to apply:** When fighting an MCP tool's input schema, check if a sibling tool's response echoes the stored form — the server's JSON serializer is authoritative where the input spec is ambiguous.

## Bicep and infra

- Azure subscription App Service quota can be 0 per-region across ALL tiers (Dynamic/Basic/Standard). **Why:** Sandbox / cost-controlled subscriptions often have hard-capped App Service VM quotas. The CTA sub has 0 quota in East US for every App Service tier, but existing plans in East US 2 imply that region has quota. **How to apply:** Before picking a region for a new Function App deployment, either (a) check with `az appservice plan list` which regions already have active plans on the subscription, or (b) run `az deployment group what-if` to catch quota issues before the real deploy. Do not assume "my favorite region" works.
- `az bicep build` produces a nested ARM JSON file as `main.json` alongside `.bicep` templates when using `--outdir` — leftover files can pollute git. **Why:** The JSON output is a build artifact, not source. **How to apply:** `.gitignore` includes `main.json` and `*.json.bak` for this reason. Never commit ARM JSON that was compiled from Bicep.
- `dependsOn` in Bicep is needed when a child resource references a parent resource via `existing` AND the parent is created by another module call. **Why:** Bicep's implicit dependency analyzer only follows direct resource references, not cross-module `existing` lookups. **How to apply:** When a module calls `existing` on a resource that was created by another module, either (a) add an explicit `dependsOn` in the parent `main.bicep` module call, or (b) pass the output of the creator module as a param to avoid the `existing` lookup.
- Deployment `what-if` catches App Service quota issues before real deploy attempts. **Why:** Quota violations surface during validation, not just provisioning. A 5-second `what-if` saves 5 minutes of partial deployment rollback. **How to apply:** Always run `az deployment group what-if` before `create` on new SKU/region combinations. Especially important for resources with per-region quotas (App Service, VMs, managed identities).
